## Causa raiz (agrupada)

### CQ-F24-001 — Misleading projector name

`handleUserDeleteJobTerminal` chama `projectUserCreateIntoReadModel()` — nome sugere só create, mas projector roteia create e delete corretamente.

### CQ-F24-002 — Unused `$attribute` parameter

`TenantGroupNameRules::forAttribute(string $attribute)` ignora o parâmetro; API (`name`) e painel (`groupName`) compartilham o mesmo array estático sem diferenciação de atributo nas regras.

### INT-F24-001 — User poll wipes group message

`pollPendingUserJob()` invoca `pollPendingGroupJob()` e depois handlers de usuário com `clearMessages()` incondicional — mensagem de grupo some se job de usuário termina no mesmo tick de 3s.

### CQ-F24-003 — Dual-success group overwrite

`preserveMessages` evita `clearMessages()` mas `successMessage = $successText` ainda sobrescreve a mensagem anterior quando create e delete de grupo terminam com success no mesmo tick.

## Estratégia de correção

### F25.1 — Rename projector helper

Renomear `projectUserCreateIntoReadModel` → `projectUserJobIntoReadModel`; atualizar call sites em `handleUserCreateJobTerminal` e `handleUserDeleteJobTerminal`. Sem mudança de comportamento.

### F25.2 — Meaningful `$attribute`

Usar `$attribute` nas regras Laravel (prefixo `{$attribute}.` onde aplicável) e/ou nas mensagens de validação customizadas; manter corpo de regras compartilhado (regex, max, admin closure). Atualizar `TenantGroupNameRulesTest` se necessário.

### F25.3 — Message accumulation

Helper de acumulação/append para `successMessage`/`errorMessage`; estender padrão `preserveMessages` aos handlers de usuário quando poll de grupo produziu mensagem no mesmo tick; Pest para cross-job e dual-success.
