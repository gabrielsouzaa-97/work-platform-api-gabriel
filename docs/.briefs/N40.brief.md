## Quality Constraints

1. **FK `customer_slug`, não `customer_id` (N40.1)** — `Customer` usa PK string `slug` (`app/Models/Customer.php:18-22`; migration `2026_05_08_000004_create_customers_table.php:14`). Padrão do projeto: `jobs.customer_slug`, `idempotency_keys.customer_slug`, `audit_logs.resource_id`. Migration `tenant_users` DEVE ter `customer_slug varchar(64) FK → customers.slug` + `unique(customer_slug, username)`. PK uuid na row `tenant_users`; índice em `customer_slug`.

2. **Hook do projector fora da transação e após guards (N40.2)** — Em `WebhookHandler::applyFinishedEvent` (`app/Modules/Jobs/Services/WebhookHandler.php:124-234`):
   - **NÃO** projetar quando `$job->state === $canonical && ! $isSummaryRecoveryRetry` (early return L134-136) — webhook re-entregue idempotente.
   - **NÃO** projetar em `$isSummaryRecoveryRetry` (early return L203-205) — só recupera `summary`.
   - Projetar **somente** quando `! $isSummaryRecoveryRetry` **e** `$canonical === 'success'`.
   - Disparar `TenantUserProjector` **após** `DB::transaction()` (mesmo padrão de `ProbeCustomerReadinessJob::dispatch` L227-229), nunca dentro da transação — evita projeção órfã se rollback.
   - Usar `$job->cmd_canonical` (não só `job_type`) via `JobTypeTranslator::jobTypeToCmd()` se necessário — campos coexistem (`app/Models/Job.php:25-26`).

3. **Mapeamento de eventos do projector (N40.2)**:
   - `cmd_canonical === 'users:create'` + success → upsert por `(customer_slug, username)` com `username` de `payload_sanitized['args'][0]` (`LifecycleAsyncAction.php:246`).
   - `cmd_canonical === 'users:delete'` + success → delete row por `(customer_slug, username)` de `args[0]`.
   - `job_type === 'provision'` + success + transição `PROVISIONING_FINISHING` (L181-199) → upsert username `admin`, `origin=provision`, groups mínimo `['admin']` (conta plataforma; governança ISSUE-050).
   - `job_type === 'deprovision'` + success → purge todas rows `tenant_users` do `customer_slug` (customer soft-deleted L183-194).
   - Gate em `$canonical === 'success'` apenas — **não** confiar em `exit_code` do webhook (ISSUE-013: callbacks frequentemente `exit_code=null`).

4. **Nunca persistir senha nem stdin sensível (N40.2, N40.5)** — `payload_sanitized` já exclui stdin (`LifecycleAsyncAction.php:246` — só `cmd`+`args`). Projector **proibido** de ler `AuditLog.payload` de `*_initiated` nem enriquecer de logs SSH (`summary`). Campos permitidos na projeção: `username`, `email`, `quota`, `groups`, `origin`, `synced_at`. Se email/groups forem necessários no upsert webhook, enriquecer `payload_sanitized` **no enqueue** com campos não-secretos do `UserCreateStdinPayload` (sem `password`) — ver PB-N40-002.

5. **Idempotência do projector (N40.2)** — Upsert via `updateOrCreate(['customer_slug','username'], …)` ou equivalente. Re-entrega de webhook com mesmo estado terminal não deve duplicar nem falhar. `synced_at` atualizado só em paths de sync/reconciliação, não em projeção webhook (distinguir `origin`).

6. **Sync service espelha `CustomerSyncService` (N40.3)** — Padrão: command `tenant-users:sync {--customer=}` iterando customers `status=active` (ou por cluster), `TenantUserSyncService` chama `OccPassthroughService::exec($customer, 'user:list', ['--json'])` (timeout explícito). Reconciliar insert/update/delete vs projeção local. Registrar drift em `AuditLog` com actions dedicadas (`tenant_user_drift_*`). Agendar em `routes/console.php` daily (referência `CustomersSyncCommand` L20 — offset sugerido `03:15` para não colidir SSH com `customers:sync` L20).

7. **Drift sem falso-positivo para admin plataforma (N40.3)** — Usuário `admin` com `origin=provision` **é esperado** no grupo `admin` — **não** emitir drift `client_in_admin_group`. Drift `client_in_admin_group` aplica-se a `username !== 'admin'` OU `origin !== 'provision'` com grupo `admin`/`subadmin`. Drift `orphan_upstream_user`: usuário em `user:list` sem row local e sem ser conta plataforma conhecida.

8. **OccPanel lê projeção; SSH só no sync on-demand (N40.4)** — `loadUsers()` (`OccPanel.php:255-268`) hoje faz SSH síncrono 30s — substituir por query `TenantUser` por `customer.slug`. Botão "Atualizar" chama `TenantUserSyncService::syncCustomer($customer)` + reload local. Manter `$usersLoading`/`$usersError`. `pollPendingUserJob` em success (`L472-474`) deve recarregar da projeção, **não** SSH — mas projector webhook pode atrasar ms; aceitável recarregar projeção (eventual consistency) ou re-fetch após poll se row ainda ausente (retry leve, sem SSH no happy path).

9. **Enforcement política admin (N40.5)** — Bloquear grupo `admin` em:
   - `OccPanel::createUser` — validar `$userGroups` comma-separated **e** rejeitar `admin` case-insensitive (complementa bloqueio username L284-286).
   - `CreateUserRequest` (`app/Http/Requests/Lifecycle/CreateUserRequest.php:38-43`) — rejeitar `groups.* === 'admin'` **e** `subadmin_groups.* === 'admin'` (API v1 herda via `V1\CreateUserRequest`). Bloquear username `admin` também na API (hoje só painel N39.2).
   - Não confundir `subadmin_groups` com membership — ambos são vetores de privilégio NC.

10. **Testes e backfill (N40.3, N40.4)** — `OccPanelTest.php:554-664` mocka SSH em `loadUsers` — refatorar para seed `tenant_users` + assert sem `SshClientInterface::run` na abertura da aba. Tenants pré-N40 sem rows: primeira abertura pode listar vazio até sync; command daily faz backfill. Documentar em N40.6.

## Technical Advisory

1. **Lacuna de metadados no webhook `users:create`** — `payload_sanitized` persiste só `args` (username posicional). Email, quota e groups do `UserCreateStdinPayload` existem só no stdin SSH (`CustomerLifecycleController.php:36-43`, `OccPanel.php:309-313`) e **não** chegam ao projector via webhook. Sem enriquecimento no enqueue, projeção pós-create fica incompleta até próximo `tenant-users:sync` — UX degradada na tabela pós-poll (N39.3).

2. **Duas origens de conta admin** — Provision upstream cria `admin` (registrar em provision success). `OnboardingSaga::dispatchCreateAdmin` (`OnboardingSaga.php:227-250`) dispara `users:create` com username de `admin_payload` (pode ser `admin.{slug}`). Projector deve suportar ambos sem drift falso-positivo; política ISSUE-050 foca em **clientes** sem admin — conta plataforma é exceção documentada.

3. **Janela eventual consistency pós-poll** — `handleUserCreateJobTerminal` (`OccPanel.php:467-479`) roda quando job fica terminal; projector roda no mesmo webhook handler. Ordem: webhook transaction → projector dispatch → poll Livewire. Em teoria mesma request cycle; em produção poll é 3s depois — row deve existir. Testar race com projector síncrono pós-transaction.

4. **`deleteUser` sem poll (N39 gap)** — `deleteUser` (`OccPanel.php:357-375`) só enfileira; lista não atualiza até refresh manual. N40.4 pode adicionar reload pós-sucesso síncrono da projeção ou poll opcional — fora do escopo mínimo mas UX inconsistente vs create.

5. **ISSUE-013 / sync de segurança** — Webhooks com campos nulos motivam sync periódico (ISSUE-050 §Relacionado). `tenant-users:sync` é rede de segurança, não substituto de projeção webhook correta.

6. **Cross-repo** — Política "nenhum cliente admin NC" exige alinhamento upstream/provision (ISSUE-050 §6). API enforcement sozinha não impede SSH manual no host — drift detection + runbook ops.

7. **Carga SSH** — Eliminar SSH no `setTab('users')` reduz latência 2–10s (motivação ISSUE-050). Sync daily + on-demand concentra SSH — monitorar timeout 30s em clusters grandes.

## O que está BOM

- `payload_sanitized` já exclui senha do Job (`LifecycleAsyncAction.php:246`) — base correta para projeção segura.
- Padrão maduro de sync/reconciliação em `CustomerSyncService` + `TenantReplicaSynchronizer` + `CustomersSyncCommand` — espelhar estrutura reduz risco arquitetural.
- `normalizeUserListRows` / `normalizeUserRow` (`OccPanel.php:485-518`) — reutilizável no sync parser.
- Bloqueio username `admin` no painel (N39.2) e gate `USER_OPS_BLOCKED` (`LifecycleAsyncAction.php:32-116`) — enforcement de readiness já existe.
- Scheduler centralizado em `routes/console.php` — adicionar `tenant-users:sync` é mecânico.

## Top 3 ações prioritárias

1. Corrigir schema N40.1 para `customer_slug` FK + unique `(customer_slug, username)`.
2. Enriquecer `payload_sanitized` com metadados não-secretos do create no enqueue OU aceitar sync imediato pós-create no poll.
3. Implementar projector pós-transação com guards explícitos (skip summary-recovery + idempotent terminal).
