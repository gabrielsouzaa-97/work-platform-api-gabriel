## Causa raiz (agrupada)

### ARQ-C1 — Slug bloqueado após readiness timeout (CRITICAL)

Timeout seta `status=failed` sem soft-delete; unique slug impede retry. Fix F11 só cobria `provision.failed` webhook. **Política**: manter `failed` no timeout; liberar slug só após deprovision success + soft-delete.

### ARQ-C2 — UI sem Remover para `failed` (HIGH)

`RemoveCustomerAction` aceita `failed|provisioning_finishing`; painel só mostra Remover para `active|provisioning`.

### ARQ-C3 — Poll-stuck sem efeitos colaterais (HIGH)

`JobsPollStuckCommand` atualiza job mas não propaga customer status, probe, projectors nem saga — webhook perdido = tenant preso.

### ARQ-C4 — Sem promote manual (MEDIUM)

Tenant em `provisioning_finishing` sem escape se gate desatualizado; falta `customers:promote` operacional (≠ `TenantResumeAction` WHMCS).

### ARQ-C5 — Saga Running após customer failed (MEDIUM)

`markTimedOut` não terminaliza onboarding em `WaitReadiness`.

## Estratégia de correção

### F26.1 — UI Remover

Estender condição em `show.blade.php:46` para incluir `failed` e `provisioning_finishing`. Testes em `CustomerShowTest`.

### F26.2 — Deprovision soft-delete

Em `WebhookHandler`, branch `deprovision + success`: após `status=removed`, chamar `delete()`. Atualizar teste que documentava ausência de soft-delete. Teste retry slug pós-cleanup.

### F26.3 — Onboarding fail on timeout

Em `markTimedOut`, localizar onboarding por `tenant_slug` + `WaitReadiness` + `Running|Pending`; chamar `OnboardingSaga::markStepFailed(..., WaitReadiness, 'customer_readiness_timeout')` ou método dedicado `failWaitReadinessForSlug`.

### F26.4 — Poll → WebhookHandler

Após poll terminal, invocar `WebhookHandler::handle` com payload synthetic `job.finished` (reutilizar StateTranslator). Idempotência herdada do handler.

### F26.5 — customers:promote

Novo command: valida `provisioning_finishing` → `active` + AuditLog `customer_promoted_manual`. Exit non-zero para outros status.
