## Causa raiz (agrupada)

### Evidência real

`POST /api/v1/tenants` com payload default `{}` ou `apps: ["mail"]` → 202 → infra viva → `failed` após ~25min. O `TenantReadinessGateChecker` legado exige `mework360_memail` + `me360_theme` + SSO — apps ausentes do suite catalog. Nenhum dos 5 entry points cruza payload resolvido com requisitos do gate.

### ARQ-A1 — Nenhum entry point valida vs readiness gate (HIGH)

Validação de formato/plano/catálogo sem cruzar `apps`/`image_mode` com gate legado.

### ARQ-A2 — Plano com `plan_apps` vazio (HIGH)

`PlanAppResolver` retorna `[]` sem erro; provision aceita lista vazia.

### ARQ-A3 — Onboarding sem flags; enable_apps tarde (HIGH)

`CreateOnboardingRequest` não expõe `image_mode`/`suite_catalog`/`plan_slug`; `enable_apps` após `wait_readiness`.

### ARQ-A6 — Painel Create bypassa resolver (MEDIUM)

Livewire constrói `ProvisionPayload` direto sem `PlanAppResolver` nem validação readiness.

### ARQ-A7 — WHMCS defaults condenados (MEDIUM)

`WhmcsProvisionService` não passa por FormRequest; defaults `suite_catalog=true`, `image_mode=false`.

### ARQ-A8 — OpenAPI documenta payload condenado (MEDIUM)

Exemplo `minimal` com `apps: [files, calendar]` inválido; sem contrato de readiness.

## Estratégia de correção

### F28.1 — Contrato compartilhado (DRY)

Novo `ProvisioningReadinessContract`:

- Constantes/métodos para apps exigidos no modo legado (`mework360_memail`, `me360_theme`) — **mesma lista** consumida por `TenantReadinessGateChecker::evaluateAppList` (refactor para delegar ao contract, sem duplicar).
- Política: `imageMode=true` → satisfeito na borda (gate reduzido).
- **Proibido**: importar `TenantReadinessGateChecker` na camada HTTP.

### F28.2 — Validador + contexto resolvido + API

`ResolvedProvisionContext` (DTO): `imageMode`, `suiteCatalog`, `fullApps`, `legacyVendor`, `resolvedApps`, `planSlug`, `clusterServerId`.

`ProvisioningReadinessValidator::validate(context): ?Violation` com código `LEGACY_READINESS_UNSATISFIABLE`, `missing_preconditions[]`, hint `image_mode=true`.

Resolver usa `PlanAppResolver` + defaults de config (mesma lógica que `ProvisionCustomerRequest`).

Integrar em `ProvisionCustomerRequest::withValidator` **após** resolução de apps (ARQ-A1, ARQ-A2).

Rejeitar casos determinísticos: `image_mode=false` + `suite_catalog=true` + apps resolvidos não contêm suíte me360; plano resolve `apps=[]`; default `{}`.

### F28.3 — Livewire + WHMCS

`Create.php`: antes de `ProvisionCustomerAction::execute`, resolver apps via `PlanAppResolver` + chamar validator (mesmo caminho API).

`WhmcsProvisionService`: construir contexto resolvido com defaults explícitos + validator; 422/exception estruturada antes dispatch.

### F28.4 — Onboarding

`CreateOnboardingRequest`: expor `tenant.image_mode`, `tenant.suite_catalog`, `tenant.plan_slug` (opcionais com defaults).

Validar `apps_enabled` via `SuiteCatalogValidator` + readiness validator na borda.

`OnboardingSpec` + `OnboardingV1Controller::toOnboardingSpec`: propagar flags.

Documentar que `enable_apps` pós-readiness é insuficiente para gate legado — validação na borda bloqueia cedo.

### F28.5 — OpenAPI

Corrigir exemplos `minimal` para payload válido (`image_mode: true` ou apps/plano compatíveis).

Documentar seção readiness contract: legado vs image-mode; código 422 `LEGACY_READINESS_UNSATISFIABLE`.

Enforcar `apps` × `full_apps` mutual exclusivity no request se ainda ausente.
