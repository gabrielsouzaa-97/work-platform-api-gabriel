## Causa raiz (agrupada)

### CQ-F23-001 — DRY group name rules

Regras de validação de nome de grupo (regex, max, `admin` reservado) existem em `CreateGroupRequest` e `OccPanel::groupNameRules()` — comportamento correto, manutenção frágil.

### CQ-F17-001..003 — groups semantics

- **001**: `groups: []` só chega ao upstream via override pós-`build()` — `UserCreateStdinPayload` omite array vazio.
- **002**: OccPanel sem teste de override vazio (API cobre).
- **003**: `groups: null` no JSON vira `[]` explícito em vez de herdar template.

### CQ-F23-002/003 — sync metric + poll UX

- **002**: `updated` só incrementa quando `inserted === 0` — subreporta em sync misto.
- **003**: `pollPendingGroupJob` processa create+delete no mesmo tick; `clearMessages()` no segundo terminal apaga o primeiro.

### N40 LOW backlog

- **CQ-N40-003**: `deleteUser` não chama `loadUsers()` pós-success (assimetria com create poll).
- **QA-N40-003**: falta teste webhook out-of-order (`job.started` após terminal) com `tenant_users` intacto.
- **QA-N40-004**: teste poll create usa seed manual, não caminho webhook→projector→poll.

## Estratégia de correção

### F24.1 — TenantGroupNameRules

Extrair `TenantGroupNameRules::forAttribute()` e consumir em API + Livewire. Sem mudança de comportamento.

### F24.2 — Sync metric

Sempre incrementar `updated` por row existente refrescada (ou incrementar no loop), independente de inserts no mesmo batch.

### F24.3 — Poll messaging

Opções: acumular mensagens, um job por tick, ou handlers `wire:poll` separados — evitar `clearMessages()` destruir feedback do job irmão.

### F24.4 — Groups stdin + null semantics

Centralizar emissão de `groups: []` no payload builder; API distingue omitido/null de array vazio explícito; Pest OccPanel para override vazio.

### F24.5 — SuiteCatalogPathResolver tests

Unit Pest com `config()` stub + arquivos temp em `storage/app/`: first-readable-wins, path inexistente, `RuntimeException` quando todos ilegíveis.

### F24.6 — deleteUser reload

Espelhar padrão create: após terminal success de delete, `loadUsers()` (ou poll equivalente).

### F24.7 — Integration tests N40

- Out-of-order: terminal success → depois `job.started` → projeção inalterada.
- Poll: Lifecycle/webhook → projector → poll success sem seed manual de `tenant_users`.
