# Plano de Arquitetura v0 — Dois Contratos (API externa OpenAPI + protocolo NC interno)

> Autor: Arquiteto sênior (ETAPA 1 do painel adversarial — `tests/arch-panel.md`)
> Repositório-âncora: `work-platform-api` (Laravel 12 control plane)
> Objetivo de referência: `tests/arch-objetivo.md`
> Status: v0 (DECIDIDO, sujeito à crítica do painel)

---

## 1. Decisão em uma frase

Introduzir um **PlatformPort** (interface de domínio único) como o **Anti-Corruption Layer** entre o control plane e a plataforma Nextcloud, e expor para fora um **contrato OpenAPI versionado `/api/v1/*`** modelado em capabilities de negócio (Tenant, Branding, Apps, Users, Onboarding, Job). Tudo que hoje conhece OCC/exit-code/`manage.sh`/staging passa a viver **somente atrás do port**; o `/occ/*` cru é **despublicado do contrato externo** e rebaixado a uma superfície interna/admin. A separação é incremental e mapeada 1:1 nas 3 fases do Platform V2 (SSH → Farm Agent → handlers tipados), de modo que trocar o transporte interno nunca toque o contrato externo.

---

## 2. Contexto atual ancorado no código real

O código confirma que a separação **já começou organicamente** — falta consolidá-la e formalizá-la.

**O que existe e funciona (preservar):**

- `ProvisionCustomerAction` / `RemoveCustomerAction` — provisionamento async com idempotency-key 24h (`IdempotencyKey`), callback HMAC (`config('app.url').'/api/jobs/hook?cluster='`), branding via stdin (`MAX_PAYLOAD_STDIN_BYTES = 262144`) ou SFTP staging (`--staging-id`). Já são **Application Services** de fato.
- `LifecycleAsyncAction` — `users:*`, `groups:*`, `apps:*` → `202 {job_id}`. Já centraliza idempotência, readiness gate (`TenantNotReadyException` → 503 + `Retry-After`) e tradução de verbo.
- `JobTypeTranslator` — **três** vocabulários já isolados: `cmd_canonical` ↔ `job_type` ↔ argv CLI upstream (`CMD_TO_CLI_ARGV`). É o embrião do ACL e a peça-chave que deve crescer para dentro do port.
- Farm Agent (N19/Fase 1): `AgentTransportResolver::shouldUseAgentTransport()` + `AgentUpstreamGateway::runAsync()` já roteiam SSH vs Agent **por cluster** dentro do `ProvisionCustomerAction`. O dual-transport existe — mas hoje vazado dentro da Action, não atrás de um port.
- `SshClientInterface` — abstração de transporte SSH (run, runAsync, inboxInit, sftpUpload, ping). É o adapter de baixo nível, mas **não** é o port de domínio (é orientado a `nextcloud-manage`, não a capabilities).
- Webhook receiver HMAC + IP allowlist (`VerifyWebhookHmac`), máquina de estados do customer, auditoria LGPD (`AuditLog`).

**As 4 dores, com evidência no código:**

1. **Vazamento de protocolo NC ao cliente.** `OccController::runOccExec()` traduz `SshRemoteException(exit 16)` em `403 { error: occ_subcmd_not_allowed, subcmd, exit_code: 16 }` — expõe nome de subcmd OCC e exit-code upstream no corpo da resposta externa. `setQuotaAll`/`filesRescan` devolvem `501 { occ_subcmd_not_supported, detail: "...occ-exec...allowlist..." }`. O cliente externo recebe semântica do Nextcloud, não do meWork360.
2. **Drift de contrato (DOC-001/CQ-F5-001).** `docs/openapi.yaml` (v2.3) promete envelope `{ success, message, data }` (linhas 27-28), mas o código retorna `CustomerResource` (JsonResource cru), `{ job_id }` e `{ error }`. O spec descreve uma API que não existe.
3. **Orquestração jogada ao cliente.** Não há operação composta. Quem integra encadeia `POST /customers` → polling `/queue/{id}` → `tenant_not_ready` 503 → `users` → `apps/enable` → branding manualmente. `docs/PLATFORM-V2-PLAN.md §2.5` já descreve esse fluxo como sequência de chamadas.
4. **Acoplamento espalhado (~52 chamadas OCC/SSH).** `OccPassthroughService`, `CustomerReadinessProbe` (`occ-exec user:list`), as duas Actions e o `AgentUpstreamGateway` cada um monta seu próprio argv `nextcloud-manage`. Não há um ponto único que conheça o protocolo.

## 3. Abordagem recomendada (decidida)

### 3.1 Topologia — quatro camadas, um port

```
            ┌──────────────────────────── CONTRATO EXTERNO (estável, versionado) ────────────────────────────┐
            │                                                                                                  │
 Cliente ──►│  ExternalController  (App\Http\Controllers\Api\V1\*)                                             │
 WHMCS      │    • routes/api_v1.php  → prefix /api/v1                                                          │
 Onboarding │    • FormRequest (validação) + ApiResource (envelope estável)                                    │
 Parceiro   │    • DomainError → HTTP estável (tenant_not_ready, branding_unavailable, retry_after)            │
            │            │  capability DTO (TenantId, BrandingSpec, AppSet, OnboardingSpec...)                  │
            │            ▼                                                                                      │
            │  ApplicationService  (App\Modules\*\Services + Actions existentes)                                │
            │    • ProvisionCustomerAction, LifecycleAsyncAction (reaproveitados)                               │
            │    • OnboardingSaga (novo) — orquestra capabilities, idempotência, compensação                    │
            └────────────┼─────────────────────────────────────────────────────────────────────────────────────┘
                         │  fala SOMENTE em capabilities de domínio (nunca subcmd OCC)
                         ▼
            ┌──────────────────────────── CONTRATO INTERNO ("protocolo NC") ─────────────────────────────────┐
            │  PlatformPort  (App\Modules\Platform\Contracts\PlatformPort — interface)                          │
            │    createTenant(TenantSpec): JobRef   removeTenant(...)   createUser(...)   enableApps(...)       │
            │    setBranding(...)  setQuota(...)  toggleMaintenance(...)  probeReadiness(): ReadinessReport     │
            │    execOcc(OccCommand): OccResult   // único ponto que conhece OCC/exit-code/allowlist            │
            │            │                                                                                      │
            │   ┌────────┴───────────────────────────────┐                                                     │
            │   ▼                                          ▼                                                     │
            │  SshPlatformAdapter                         AgentPlatformAdapter                                  │
            │   (envolve SshClientInterface)               (envolve AgentUpstreamGateway / AgentCommandQueue)    │
            │   conhece argv `nextcloud-manage`,           conhece envelope JSON tipado (schema_version,         │
            │   occ-exec, staging, exit-codes              operation, operation_id, callback_token)             │
            └────────────┼──────────────────────────────────────────┼──────────────────────────────────────────┘
                         ▼                                          ▼
                  SSH `nextcloud-manage --async`            POST /api/agent/v1/commands (Farm Agent outbound)
                  + webhook HMAC /api/jobs/hook             + /events (progresso por step)
```

Decisões estruturais:

- **`PlatformPort` é a fronteira do ACL.** Assina em capabilities (`createTenant(TenantSpec): JobRef`), nunca em subcmds. O `JobTypeTranslator` migra para dentro do adapter SSH como tradutor capability→argv. `AgentTransportResolver` deixa de ser chamado dentro da Action e passa a ser a **fábrica que escolhe o adapter** (`PlatformPortFactory::for(ClusterServer)`).
- **Adapters são intercambiáveis por cluster.** A lógica `shouldUseAgentTransport()` que hoje vive embutida no `ProvisionCustomerAction` (linhas 143-162) sai da Action e vira responsabilidade da factory. A Action passa a depender só de `PlatformPort`.
- **Erros tipados de domínio.** Um `DomainError` enum/hierarquia no contrato externo (`TenantNotReady`, `BrandingUnavailable`, `CapabilityNotSupported`, `UpstreamUnavailable`) com mapeamento único `DomainError → HTTP` num exception handler dedicado da v1. Os adapters traduzem `SshRemoteException`/exit-code para `DomainError` **dentro do port** — o exit-16 nunca cruza a fronteira.

### 3.2 Contrato EXTERNO — namespace `/api/v1`

- **Novo prefixo de rotas `routes/api_v1.php`** sob `Route::prefix('v1')`, controllers em `App\Http\Controllers\Api\V1\`. As rotas atuais (`/api/customers`, `/api/customers/{slug}/occ/*`, `/api/customers/{slug}/users`) **permanecem intactas** (legado/admin interno) — zero big-bang.
- **Envelope estável decidido:** padronizar a v1 no formato que o código **já produz** para escrita assíncrona — `202 { job_id, ... }` e `JsonResource` para leitura — e **corrigir o spec para refletir o código** (não o contrário). Refutamos reescrever o código para o envelope `{ success, message, data }` documentado: isso é trabalho grande sem valor de negócio e contradiz o que os clientes (onboarding-api) já consomem. A fonte de verdade externa passa a ser a v1, modelada a partir do comportamento real.
- **Capabilities de negócio** (não subcmds): `POST /v1/tenants`, `DELETE /v1/tenants/{slug}`, `POST /v1/tenants/{slug}/users`, `POST /v1/tenants/{slug}/apps`, `PUT /v1/tenants/{slug}/branding`, `PUT /v1/tenants/{slug}/quota`, `GET /v1/jobs/{id}`, `POST /v1/onboarding`.
- **Versionamento e deprecação:** `Sunset` HTTP header + `Deprecation` para rotas legadas; política "N e N-1 suportadas; deprecação anunciada com 90 dias".

### 3.3 Destino do `/occ/*` (DECISÃO)

**Despublicar do contrato externo público.** O `/occ/*` é protocolo NC vazado. Concretamente:

1. As capabilities **estáveis** que hoje passam por OCC e funcionam (branding por chave, quota por usuário, app:enable, files:scan por usuário) são re-expostas como capabilities de domínio na v1 (`PUT /v1/tenants/{slug}/branding`, `PUT /v1/tenants/{slug}/quota/{user}`), implementadas **via PlatformPort** (`setBranding`/`setQuota`/`enableApp`).
2. O `OccController` cru permanece **apenas** sob a malha legada/admin (auth `active.operator` + permissão de operador interno), nunca no client SDK externo, nunca documentado no `openapi-external.yaml`. É a "válvula de escape" de baixo nível para operação.
3. Capabilities hoje quebradas por allowlist upstream (D-02: `user:setting --all`, `maintenance:mode`, `config:app:set`) **não entram na v1 enquanto o upstream não as suportar**. A v1 retorna `404 capability_not_available` (limpo, sem `subcmd`/`exit_code`) ou simplesmente não expõe a rota — o gap de capability é resolvido no `work-platform-scripts`/Farm Agent (operação tipada `occ.exec` allowlisted, Platform V2 §2.4), não maquiado na borda.

### 3.4 Operações compostas — `POST /v1/onboarding` (saga)

Prova do modelo: um `OnboardingSaga` (Application Service) que orquestra capabilities via `PlatformPort` e expõe status por etapa.

- **Modelo de execução:** o saga **não** é síncrono. Retorna `202 { onboarding_id }` imediatamente; `GET /v1/onboarding/{id}` expõe cada etapa (`tenant_create`, `admin_create`, `apps_enable`, `branding`) com estado individual — espelhando os steps canônicos do Platform V2 (`accepted → db_created → containers_up → apps_installed → ready`). Refutamos prometer onboarding síncrono: o provisionamento real leva 5-15 min (ISSUE-010 readiness).
- **Idempotência:** chave de idempotência de negócio no header (`Idempotency-Key`), reaproveitando a infra `IdempotencyKey` 24h já existente, com hash do `OnboardingSpec`.
- **Compensação/rollback parcial:** cada etapa registra estado persistido; falha em etapa N **não** desfaz etapas concluídas automaticamente por padrão (rollback de tenant provisionado é caro e arriscado). Decisão: **compensação manual assistida** — o status expõe a etapa que falhou e oferece retry idempotente daquela etapa; só `tenant_create` falho dispara remoção automática do tenant-fantasma (reaproveita o fluxo de ghost/soft-delete que `ProvisionCustomerAction` já implementa).

### 3.5 Gestão dos dois specs + testes de contrato

- **Fonte de verdade EXTERNA:** `docs/openapi-external.yaml` (novo), cobrindo **somente** `/api/v1/*`. O `docs/openapi.yaml` atual é congelado e marcado "internal/legacy".
- **Contrato INTERNO:** o `PlatformPort` PHP **é** o contrato (interface tipada + DTOs). Para o envelope Farm Agent (JSON tipado do §2.3 do Platform V2), versionar `docs/schemas/agent-operations.v1.json` (JSON Schema) e validar em ambos os lados.
- **Geração code-first decidida:** gerar o `openapi-external.yaml` **a partir do código** (atributos PHP / `dedoc/scramble` ou similar) em vez de escrever YAML à mão — é a causa raiz do drift DOC-001 (spec escrito à mão diverge). Trade-off aceito: dependência de tooling.
- **CI que falha no drift:** (a) job que regenera o spec e faz `git diff --exit-code` (spec desatualizado quebra o build); (b) testes de contrato (`spectator` / schema assertion) validando que cada response da v1 casa com o schema; (c) validação do JSON Schema das operações Farm Agent contra fixtures. Isso fecha DOC-001/CQ-F5-001 estruturalmente.

### 3.6 Migração sem big-bang (alinhada às 3 fases do Platform V2)

| Fase V2 | O que o transporte interno faz | O que muda no contrato externo |
|---------|-------------------------------|-------------------------------|
| **Fase 1 (N19, feito)** — SSH→Agent transporte | `PlatformPort` tem 2 adapters; agente chama `manage.sh` por baixo | **Nada.** v1 já fala só com o port |
| **Fase 2 (N20)** — operações tipadas | `AgentPlatformAdapter` usa `tenant.create`/`memail.configure` tipados; OccAdapter único | **Nada.** capabilities iguais; só o adapter muda |
| **Fase 3 (N23/N24)** — DNA + placement | factory escolhe farm por capacidade, não 1:1 cluster | **Nada** externo; surge `placement` opcional no `OnboardingSpec` |

A garantia central: **o contrato externo não tem dependência de transporte**. Trocar SSH por Agent por handler tipado é uma troca de adapter atrás do port.

## 4. Roteiro em fases (com critérios de sucesso mensuráveis)

> Cada fase é uma sprint categoria N/F no pipeline `/pmo`. Sem big-bang: legado intacto até a v1 atingir paridade.

**Fase A — Extrair o PlatformPort (refactor sem mudança de comportamento).**
Criar `App\Modules\Platform\Contracts\PlatformPort` + `SshPlatformAdapter` (envolve `SshClientInterface`+`JobTypeTranslator`) + `AgentPlatformAdapter` (envolve `AgentUpstreamGateway`) + `PlatformPortFactory`. `ProvisionCustomerAction`/`LifecycleAsyncAction`/`OccPassthroughService`/`CustomerReadinessProbe` passam a depender do port.
*Sucesso:* zero chamada direta a `SshClientInterface`/`AgentUpstreamGateway` fora de `app/Modules/Platform/Adapters/` (grep gate em CI); 100% dos testes existentes verdes sem alteração de asserts de resposta; a contagem de "pontos que montam argv `nextcloud-manage`" cai de ~52 para 1 adapter.

**Fase B — Mapear `DomainError` e parar o vazamento.**
Hierarquia `DomainError` + handler único; adapters traduzem exit-code→DomainError dentro do port.
*Sucesso:* nenhuma resposta de API contém as chaves `subcmd`/`exit_code`/stack trace (teste de contrato negativo); todas as exceções de transporte mapeadas para ≤8 erros de domínio estáveis.

**Fase C — Stand up `/api/v1` (capabilities de leitura + tenant + jobs).**
`routes/api_v1.php`, controllers V1, FormRequests, Resources; `openapi-external.yaml` gerado do código; CI de drift.
*Sucesso:* `POST /v1/tenants`, `DELETE /v1/tenants/{slug}`, `GET /v1/jobs/{id}` em produção; `git diff --exit-code` do spec gerado passa no CI; testes de contrato cobrindo 100% das rotas v1.

**Fase D — Capabilities de mutação via port + despublicar `/occ/*`.**
`branding`, `quota/{user}`, `apps`, `users` na v1; `OccController` rebaixado a admin/legacy (fora do spec externo); `Deprecation`/`Sunset` headers nas rotas antigas equivalentes.
*Sucesso:* onboarding-api migrado para `/v1/*` (consumidor real); `/occ/*` ausente do `openapi-external.yaml`; dashboard de uso mostra tráfego externo migrando para v1.

**Fase E — `POST /v1/onboarding` (saga).**
`OnboardingSaga` + status por etapa + idempotência + compensação assistida.
*Sucesso:* um signup completo (tenant→admin→apps→branding) por **uma** chamada + polling de status; idempotência verificada (replay não duplica tenant); etapa falha expõe retry sem efeito colateral.

Ordem: A → B → C → (D ∥ E parcial). A e B são pré-requisito duro; bloqueiam o resto.

## 5. Trade-offs explícitos

- **Port + adapters vs. continuar com Actions diretas.** Escolhido o port: o custo (uma camada de indireção, ~3-4 classes novas) é pago uma vez e elimina a dispersão de 52 pontos + viabiliza Fase 2/3 sem tocar a borda. O risco é over-engineering se as capabilities forem poucas — mitigado mantendo o port enxuto (só capabilities que existem hoje).
- **Spec gerado do código vs. spec escrito à mão.** Escolhido code-first: mata o drift na raiz, ao custo de tooling e de menos liberdade de descrição. Aceito porque o drift já causou DOC-001.
- **Dois specs vs. um.** Escolhido dois (externo gerado + interno PlatformPort/JSON Schema): o interno muda rápido (3 fases), o externo deve ser estável; uni-los acoplaria a cadência. Custo: dois artefatos para manter — mitigado porque o interno é código, não YAML.
- **`/occ/*` despublicar vs. manter.** Escolhido despublicar do externo, manter interno: preserva a válvula de operação sem expor protocolo NC ao cliente.
- **Onboarding async vs. sync.** Escolhido async com status por etapa: prometer sync seria mentira (5-15 min de readiness). Custo: cliente precisa de polling — já é o padrão da API.

## 6. Riscos principais e mitigação

| Risco | Severidade | Mitigação |
|-------|-----------|-----------|
| **Over-engineering do port** (capabilities especulativas, generalização prematura) | Alta | Port só expõe capabilities com uso real hoje; YAGNI; revisão na Fase A |
| **Duplo spec divergente** (externo e interno saem de sincronia) | Média | Externo é gerado do código; interno é a própria interface PHP; ambos no CI |
| **Falsa promessa de sync no onboarding** | Média | Contrato 202 + status por etapa desde o design; documentar SLA de readiness |
| **Refactor Fase A introduz regressão silenciosa** | Alta | Fase A é refactor puro: nenhum teste de resposta muda; characterization tests antes de mexer |
| **Migração de consumidores** (onboarding-api, WHMCS já usam rotas antigas) | Média | Legado intacto; `Deprecation`/`Sunset`; v1 opt-in; medir tráfego antes de desligar |
| **Exit-code/allowlist continua vazando por caminho não coberto** | Alta | Teste de contrato negativo (nenhuma resposta contém `subcmd`/`exit_code`); grep gate |
| **Acoplamento do port ao `nextcloud-manage`** (port "vaza" semântica SSH) | Média | Port assina em capabilities de domínio; revisão de que nenhum método cita OCC/argv |

## 7. Resumo das decisões-chave

1. **PlatformPort** (interface única, ACL) com `SshPlatformAdapter` + `AgentPlatformAdapter`; `JobTypeTranslator` e `AgentTransportResolver` migram para dentro/para a factory do port. Consolida as ~52 chamadas em 1.
2. **Contrato externo `/api/v1`** novo namespace, capabilities de negócio, envelope **alinhado ao código real** (corrigir o spec, não o código), spec **gerado code-first** + CI de drift.
3. **`/occ/*` despublicado** do contrato externo (rebaixado a admin/interno); capabilities quebradas por allowlist não entram na v1 até o upstream suportar (resolvido no Farm Agent, não maquiado na borda).
4. **`POST /v1/onboarding`** como saga async com status por etapa, idempotência reaproveitando `IdempotencyKey` 24h e compensação assistida.
5. **Migração em 5 fases** (A extrair port → B parar vazamento → C v1 base → D mutações + despublicar occ → E saga), alinhada 1:1 às 3 fases do Platform V2, com a garantia de que **o contrato externo não tem dependência de transporte**.
